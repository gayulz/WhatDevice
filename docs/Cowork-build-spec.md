# Cowork 개발 지시서 (Build Spec) — 애플 기기 식별자 변환기

> **이 문서를 읽는 Cowork에게:** 이 문서는 완전한 개발 명세다. 아래 결정사항대로
> 처음부터 끝까지 자율로 개발하라. 사람은 자고 있으므로 중간에 질문하지 말고,
> 명세에 없는 갈림길은 **10번 항목 "막혔을 때 행동 지침"**을 따르라.
> 함께 제공된 `device-id-tool-roadmap.md`에 배경과 큰 그림이 있다.

---

## 0. 목표 (한 줄)

`iPhone15,2` ↔ `iPhone 14 Pro` 를 양방향으로 변환·검색하는 한국어 정적 웹사이트를
바닐라 JS + Java 빌드 스크립트로 만든다. 서버·DB 없음. 결과물은 정적 HTML.

## 3. 기술 스택 (확정)

- **빌드**: Java 21 + Gradle. `./gradlew run` 한 번으로 `/dist` 전체 생성.
- **의존성**: Jackson(JSON 파싱) 외 최소화. 템플릿 엔진 금지 — `String.replace()`로 치환.
- **화면**: 순수 HTML + CSS + 바닐라 JS. 외부 JS 프레임워크/번들러 금지.
- **호스팅 전제**: `/dist`를 그대로 정적 호스팅. 경로는 상대경로 사용(서브경로 배포 대비).

## 4. 데이터 처리 규칙

### 4.1 원본
- URL: `https://raw.githubusercontent.com/clo4/apple_device_identifiers/main/devices.json`
- 형태: `{ "제품명": "식별자" }` (값은 단일 문자열). 빌드 시 다운로드 또는 `data/`에 동봉.

### 4.2 가공
1. `iPhone`, `iPad`로 시작하는 키만 필터(기본 범위). 카테고리는 상수로 분리해 토글 쉽게.
2. 역방향 맵 생성: 식별자 → 기종명.
3. 슬러그: `iPhone15,2` → `iphone15-2` (콤마·공백 → 하이픈, 소문자).
4. `data/overrides.json`을 원본에 **병합**(override가 우선).

### 4.3 overrides.json — 누락분 보강 (⚠️ 정확성 최우선)
clo4 원본은 최신 기기가 빠져 있다. 아래 **확인된 값**을 overrides.json에 넣어라:

```json
{
  "iPhone 16": "iPhone17,3",
  "iPhone 16 Plus": "iPhone17,4",
  "iPhone 16 Pro": "iPhone17,1",
  "iPhone 16 Pro Max": "iPhone17,2"
}
```

그 이후 시리즈(iPhone 16e, iPhone 17 시리즈, 최신 iPad 등)는:
- **신뢰할 수 있는 공식/표준 소스로 조사**해서 채워라
  (Apple 지원 페이지, theiphonewiki, adamawolf gist 등).
- **확신이 없으면 추측해서 넣지 말고 비워두라.** 부정확한 데이터는 신뢰를 깎는다.
- 조사해서 넣은 값과 출처를 `DECISIONS.md`에 기록하라.

## 5. 빌드 스크립트 요구사항 (`BuildSite.java`)

처리 흐름:
1. `devices.json` + `overrides.json` 읽어 병합
2. 범위 필터(iPhone/iPad) + 역방향 맵 생성
3. 각 기기마다 `templates/device.html` 치환 → `dist/device/{슬러그}.html`
4. 검색용 `dist/data.js` 생성 (전체 목록을 JS 배열로; 식별자·기종명·슬러그·카테고리)
5. `templates/index.html` → `dist/index.html`
6. `dist/sitemap.xml` 생성 (전 페이지 URL)
7. `dist/robots.txt` 생성 (sitemap 명시)
8. `static/*` → `dist/` 복사
9. 가이드/정책 페이지 생성

코드 품질: 함수 단위로 명확히 분리, 한국어 주석, 하드코딩된 설정은 상수로 모아두기.

## 6. 페이지 / 디자인 요구사항

### 6.1 메인 (`index.html`)
- 상단: 사이트명 + 한 줄 설명
- 검색창 1개: 식별자(`iPhone15,2`)와 기종명(`아이폰 14 프로`) **양방향** 검색
- 바닐라 JS가 `data.js`에서 실시간 필터 → 결과 카드 → 클릭 시 상세 페이지
- 다크모드 토글 버튼
- 애드센스 placeholder 영역
- 하단 푸터: 정책 페이지 링크 + 면책 문구

### 6.2 기기 상세 (`device/{슬러그}.html`)
- `<title>`: `{식별자} - {기종명} | WhatDevice` (검색어를 제목에)
- H1: `{식별자} — {기종명}`
- 본문: 식별자 / 기종명 / 슬러그 / (overrides에 있으면) 출시정보
- 같은 시리즈 관련 기기 내부 링크
- FAQ 1~2개 (예: "iPhone15,2가 무슨 기종인가요?")
- 애드센스 placeholder

### 6.3 디자인
- 톤: 미니멀, 개발자 친화. 식별자는 **monospace** 폰트로 강조.
- 반응형 필수(모바일 우선). 다크/라이트 둘 다 깔끔하게.
- 색상은 차분한 단색 액센트. 시스템 폰트 스택 사용(웹폰트 최소화 → 속도).

## 7. 콘텐츠 요구사항 (한국어, Cowork가 초안 작성)

- **가이드 글 2개** (`dist/guide/...`):
  1. "애플 기기 식별자(모델 코드)란 무엇인가"
  2. "크래시 로그·애널리틱스에서 기기 모델 확인하는 법"
  - 각 800자 이상, 자연스러운 한국어, 내부 링크 포함.
- **정책 페이지 3개**: 이용약관 / 개인정보처리방침 / 문의
  - 일반적인 정적 사이트용 표준 템플릿. 문의 페이지에 2번 항목 5번의 이메일.
- ※ 모든 글은 사람이 나중에 검수한다는 전제. 사실관계는 보수적으로.

## 10. 막혔을 때 행동 지침 (핵심)

명세에 없는 결정이 필요하면 **멈추지 말고** 다음을 따른다:
1. 가장 단순하고 표준적인 선택지를 고른다.
2. 되돌리기 쉬운 방향을 택한다(나중에 사람이 바꾸기 쉽게).
3. 그 결정과 이유를 `DECISIONS.md`에 한 줄로 적는다.
4. 계속 진행한다.

절대 사용자 입력을 기다리며 작업을 중단하지 말 것. 아침에 사람이 `DECISIONS.md`를
보고 검토할 것이다.
